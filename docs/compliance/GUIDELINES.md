## 1. GDPR（一般データ保護規則）対応

### 1.1 適用範囲
- EUに拠点を持つ顧客
- EUデータ主体の個人データを処理する顧客

### 1.2 対応項目

#### データレジデンシー（第44条）
- **要件**: 個人データはEU域内で処理される必要がある
- **対応**: Self-Hosted版ではすべての個人データを顧客環境（オンプレ/VPC）に保持
- **技術的保証**: MCPペイロードはクラウドに送信しない

#### Right to Erasure（第17条）
- **要件**: データ主体は自身のデータ削除を要求できる
- **対応**:
  ```sql
  -- ユーザーデータの完全削除
  DELETE FROM mcp_server_request_log WHERE user_id = '<user_id>';
  DELETE FROM mcp_config WHERE user_id = '<user_id>';
  DELETE FROM user WHERE id = '<user_id>';
  ```
- **実装**: `DELETE /api/users/:userId/gdpr-erase` エンドポイント（Manager UI）

#### Data Portability（第20条）
- **要件**: データ主体は自身のデータを受け取る権利がある
- **対応**:
  ```bash
  # データエクスポート
  curl -X POST https://manager.tumiki.io/api/users/me/export \
    -H "Authorization: Bearer ${ACCESS_TOKEN}" \
    -o my-data.json
  ```
- **形式**: JSON形式でエクスポート

#### Data Protection by Design（第25条）
- **要件**: デフォルトでプライバシー保護を実装
- **対応**:
  - PIIマスキング（デフォルト有効）
  - 暗号化保存（Prisma Field Encryption）
  - 最小権限の原則（デフォルトViewer権限）

#### Data Breach Notification（第33条）
- **要件**: 72時間以内にデータ侵害を報告
- **対応**:
  - SIEM連携でリアルタイム検知
  - インシデントレスポンスフロー整備
  - 自動通知機能（Slack/Email）

### 1.3 DPA（データ処理契約）

顧客とTumiki間のDPAでは以下を規定：
- Tumikiはデータ処理者（Processor）
- 顧客はデータ管理者（Controller）
- Self-Hosted版では、Tumikiは個人データにアクセスしない（ゼロアクセス）

## 2. 金融庁ガイドライン対応

### 2.1 適用規制
- 金融商品取引業者等向けの総合的な監督指針
- 主要行等向けの総合的な監督指針
- 金融分野におけるサイバーセキュリティ強化に向けた取組方針

### 2.2 対応項目

#### システムリスク管理（I-3-2-2）
- **要件**: システムの安全性・安定性確保
- **対応**:
  - PostgreSQL HA構成推奨（Patroni）
  - 自動バックアップ（日次フルバックアップ + WAL）
  - ヘルスチェック実装（`/health` エンドポイント）

#### 外部委託管理（I-3-3）
- **要件**: 外部委託先の適切な管理
- **対応**:
  - Self-Hosted版では顧客環境内で完結（外部委託なし）
  - Cloud版では、DPAによる契約管理

#### サイバーセキュリティ管理（I-3-5）
- **要件**: サイバー攻撃への対策
- **対応**:
  - TLS 1.3（最低限TLS 1.2）
  - JWT認証 + API Key認証
  - レート制限（DDoS対策）
  - SIEM連携（異常検知）

#### 監査証跡（II-3-2-3-2）
- **要件**: すべての重要な操作の記録
- **対応**:
  - 全MCP通信の記録（PostgreSQL + ローカルファイル）
  - 90日以上の保持
  - 改ざん防止（WORM推奨）

#### 暗号化（II-3-2-3-3）
- **要件**: 個人データの暗号化
- **対応**:
  - 保存時暗号化（AES-256）
  - 通信時暗号化（TLS 1.3）
  - APIキー暗号化（Prisma Field Encryption）

### 2.3 FISC安全対策基準対応

| 項目 | 基準 | 対応 |
|------|------|------|
| **設備基準** | 施設（1-1） | 顧客管理（オンプレ/VPC） |
| **技術基準** | アクセス制御（3-1） | Keycloak JWT、API Key |
| **技術基準** | 暗号化（3-3） | TLS 1.3、AES-256 |
| **技術基準** | ログ管理（3-5） | PostgreSQL、ローカルファイル、SIEM |
| **運用基準** | バックアップ（4-2） | 日次フルバックアップ + WAL |
| **運用基準** | 監査（4-5） | 監査ログ、SIEM連携 |

## 3. 医療情報システムの安全管理ガイドライン対応

### 3.1 適用範囲
- 医療機関（病院、クリニック）
- 医療情報を扱うシステム

### 3.2 対応項目

#### 6.1 情報の保存（真正性）
- **要件**: 正当な権限者が操作した記録であることを確認できる
- **対応**:
  - 監査ログにユーザーID、タイムスタンプを記録
  - 改ざん防止（WORMストレージ推奨）

#### 6.2 情報の保存（見読性）
- **要件**: 情報を見読可能な状態で保存
- **対応**:
  - PostgreSQL（SQL形式で検索可能）
  - JSON Lines（人間可読形式）

#### 6.3 情報の保存（保存性）
- **要件**: 法定保存期間（5年）の保存
- **対応**:
  - 自動バックアップ（90日保持）
  - 長期保存はS3 Glacier等に移行可能

#### 6.10 アクセス制御（最小権限の原則）
- **要件**: 業務上必要な範囲でのみアクセス許可
- **対応**:
  - カスタムロール（MCPサーバーごと個別権限）
  - Read/Write/Execute権限の分離

#### 6.11 ユーザー認証
- **要件**: 二要素認証等の実施
- **対応**:
  - Keycloak OTP（TOTP）対応
  - パスワードポリシー（8文字以上、複雑性要件）

#### 6.12 暗号化
- **要件**: 個人情報の暗号化
- **対応**:
  - 保存時暗号化（AES-256）
  - 通信時暗号化（TLS 1.3）

#### 6.16 監査証跡
- **要件**: すべてのアクセス記録
- **対応**:
  - 全MCP通信の記録
  - SIEM連携（リアルタイム監視）

### 3.3 三省二ガイドライン対応表

| ガイドライン項目 | 対応状況 |
|----------------|---------|
| 真正性 | ✅ 監査ログ、改ざん防止 |
| 見読性 | ✅ PostgreSQL、JSON Lines |
| 保存性 | ✅ バックアップ、長期保存対応 |
| アクセス制御 | ✅ Keycloak、カスタムロール |
| ユーザー認証 | ✅ JWT、OTP |
| 暗号化 | ✅ TLS 1.3、AES-256 |
| 監査証跡 | ✅ 全MCP通信記録、SIEM |

## 4. PCI DSS（Payment Card Industry Data Security Standard）

### 4.1 適用範囲
- クレジットカード情報を扱う顧客

### 4.2 対応項目（計画中）

#### Requirement 3: Protect Stored Cardholder Data
- **要件**: カード情報の暗号化保存
- **対応**:
  - PIIマスキング（クレジットカード番号）
  - Prisma Field Encryption

#### Requirement 4: Encrypt Transmission of Cardholder Data
- **要件**: 通信時の暗号化
- **対応**: TLS 1.3

#### Requirement 10: Track and Monitor All Access
- **要件**: すべてのアクセス記録
- **対応**: 監査ログ、SIEM連携
