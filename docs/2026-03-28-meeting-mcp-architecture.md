## プロダクト方針

### コンセプト

企業の情報をリアルタイムでAIに渡す「MCP管理プラットフォーム」。
第一フェーズとしてMCPに特化し、将来的にA2A・決済系プロトコル等へ拡張予定。

### 主要機能

- 外部ツールとの全通信をリアルタイム監視・異常検知
- 機密情報のマスキング処理
- 従業員ロール別のMCPツールアクセス権限管理
- 動的ツール検索（コンテキスト圧迫問題の解決）
- 仮想MCP統合（複数MCPを1つのサーバーとしてAIに接続）

## アーキテクチャ変更

### 変更前（現状）

```
完全クラウド or オンプレにTumiki全体をホスティング
```

### 変更後（分離型）

```
クラウド（レイブン管理）     ｜    オンプレ（クライアント社内）
────────────────────────────────────────────────────────
レジストリ（Next.js）        ｜    MCPプロキシ（ゲートウェイ）
 - MCP設定・カタログ管理     ｜     - MCPとの実通信
 - ポリシー管理              ｜     - 詳細ログ記録
 - 統計ダッシュボード        ｜     - APIキー等の機密情報保持
 - 監査ログ（メタデータ）    ｜     - オースストークン管理
```

### データ分類

| 区分 | 保存先 | 内容 |
|------|--------|------|
| 秘密情報 | オンプレのみ | APIキー・OAuthトークン・MCPの入出力実データ |
| 社外秘情報 | クラウド | 監査ログ（メタデータ）・MCP設定・統計 |

> 参考：Kong（APIゲートウェイ）と同様の設計思想。大企業向けにはクライアント社内ネットワーク内でゲートウェイを動作させる。

### 同期仕様（クラウド⇔オンプレ）

| データ種別 | 同期方式 | 頻度 |
|-----------|---------|------|
| MCP設定データ | tRPC相互通信 | リアルタイム（変更都度） |
| 監査ログ | tRPC相互通信 | 30分おき（暫定） |

## 4月版方針（PoC優先）

### 方針転換

複雑なクラウド/オンプレ分離アーキテクチャは中長期対応とし、
**4月末PoCはデスクトップアプリ（Electron）に機能集約**する。

```
MCPクライアント（Claude Code等）
        ↓ STDIO
  積木デスクトップ（プロキシ）  ← SQLite（ローカルDB）
        ↓ STDIO
   MCPサーバー（各ツール）
```

### デスクトップアプリ仕様

| 項目 | 仕様 |
|------|------|
| フレームワーク | Electron（既存） |
| DB | SQLite（ローカル完結） |
| 通信規格 | STDIOのみサポート（SSE/HTTP Streamable不要） |
| MCP接続方式 | `.mcp.json` を積木デスクトップ向けに書き換え、名前ベースでSQLiteを検索して対象サーバーを起動 |
| ログ保存 | 監査ログ → SQLite、MCPの入出力データは現状スコープ外 |

### 監査ログ記録項目

```
- ユーザーID
- MCPサーバーID
- 通信規格（STDIO / SSE / Streamable HTTP）
- メソッド名（ほぼ tool_call）
- 入出力データバイト数
- タイムスタンプ
```

> 注：MCPで取得した実データ（JSONの中身）は記録しない。「誰がいつどのMCPを叩いたか」のみ。
