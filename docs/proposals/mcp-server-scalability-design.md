# Tumiki MCP Server スケーラビリティ設計提案

**作成日**: 2026-03-04
**ステータス**: Draft

---

## 1. 背景と課題

### 1.1 現状

Tumikiは、クラウドホスティングによる簡単なMCPサーバー接続を売りにしている。

**現在対応しているTransportType:**

| Type | 説明 | 対応状況 |
|------|------|---------|
| `STDIO` | 標準入出力での通信 | Cloud Runでラッパー必要 |
| `SSE` | Server-Sent Events | 対応済み |
| `STREAMABLE_HTTPS` | Streamable HTTP | 対応済み |

### 1.2 課題

| 課題 | 詳細 | 影響 |
|------|------|------|
| **ステートフルMCPサーバー** | OAuth認証値のファイル保持、チャット記録など永続化が必要 | クラウドのステートレス前提と相性悪い |
| **stdio変換の手間** | stdio MCPサーバーをHTTP化するための変換レイヤーが個別に必要 | 新規MCPサーバー追加のたびに手間発生 |
| **スケーラビリティ** | 対応MCPサーバー数を膨大にしたい | 現状の手動対応では限界 |

### 1.3 MCPサーバーの分類

```
┌─────────────────────────────────────────────────────────────────┐
│                    MCP Server 分類                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────┐    ┌─────────────────────┐            │
│  │   ステートレス       │    │   ステートフル       │            │
│  │   (Stateless)       │    │   (Stateful)        │            │
│  ├─────────────────────┤    ├─────────────────────┤            │
│  │ - 認証不要           │    │ - OAuthファイル保持  │            │
│  │ - APIキー認証のみ    │    │ - セッション管理     │            │
│  │ - リクエスト単位処理 │    │ - ローカルDB/ファイル│            │
│  ├─────────────────────┤    ├─────────────────────┤            │
│  │ 例:                  │    │ 例:                  │            │
│  │ - context7           │    │ - claude-ai-slack    │            │
│  │ - web-search         │    │ - gdrive (OAuth)     │            │
│  │ - fetch              │    │ - memory servers     │            │
│  └─────────────────────┘    └─────────────────────┘            │
│           ↓                          ↓                         │
│     クラウド対応容易            クラウド対応困難                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 設計案

### 2.1 案A: ハイブリッドアーキテクチャ（推奨）

**概要**: クラウドとローカルエージェントのハイブリッド構成

```
┌─────────────────────────────────────────────────────────────────┐
│                      Tumiki Cloud                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    tumiki-proxy                          │   │
│  │  ┌─────────────────┐  ┌─────────────────┐               │   │
│  │  │ ステートレスMCP  │  │ ステートフルMCP  │               │   │
│  │  │ (HTTP/SSE)      │  │ (Local Agent経由)│               │   │
│  │  │       ↓         │  │       ↓         │               │   │
│  │  │   直接接続      │  │  WebSocket経由   │               │   │
│  │  └─────────────────┘  └────────┬────────┘               │   │
│  └────────────────────────────────┼─────────────────────────┘   │
│                                   │                             │
└───────────────────────────────────┼─────────────────────────────┘
                                    │ WebSocket / gRPC
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│                   Tumiki Local Agent                            │
│              (ユーザーのローカルマシン or VPS)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │ stdio MCP    │  │ OAuth認証    │  │ ローカル     │          │
│  │ サーバー群   │  │ ファイル     │  │ ストレージ   │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │
│                                                                 │
│  対応例:                                                        │
│  - claude-ai-slack (OAuth)                                     │
│  - filesystem (ローカルファイルアクセス)                        │
│  - memory-server (永続化)                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**メリット:**
- ステートフル/ステートレス両方に対応
- ローカルファイルアクセスが必要なMCPも対応可
- 既存クラウドインフラを活かせる
- ユーザーの機密データがローカルに留まる（セキュリティ）

**デメリット:**
- ユーザーにLocal Agentのインストールを求める必要
- Local Agentのメンテナンスコスト

### 2.2 案B: Universal MCP Wrapper + カタログ方式

**概要**: 汎用stdio→HTTPラッパーを1つ作り、MCPサーバー定義はカタログで管理

```
┌─────────────────────────────────────────────────────────────────┐
│                  MCP Server Catalog (DB)                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ id: "github"                                             │   │
│  │ command: "npx"                                           │   │
│  │ args: ["-y", "@modelcontextprotocol/server-github"]     │   │
│  │ envVarKeys: ["GITHUB_PERSONAL_ACCESS_TOKEN"]            │   │
│  │ stateRequirement: "STATELESS"                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ id: "slack"                                              │   │
│  │ command: "npx"                                           │   │
│  │ args: ["-y", "@anthropics/claude-ai-slack"]             │   │
│  │ envVarKeys: ["SLACK_BOT_TOKEN"]                         │   │
│  │ stateRequirement: "STATEFUL_OAUTH"                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ... 数百のMCPサーバー定義                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 動的生成
┌─────────────────────────────────────────────────────────────────┐
│              Universal MCP Wrapper (Cloud Run)                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           Generic stdio → HTTP Adapter                   │   │
│  │                                                          │   │
│  │  1. リクエスト受信                                       │   │
│  │  2. カタログからMCPサーバー定義を取得                    │   │
│  │  3. stdio MCPサーバーをspawn                             │   │
│  │  4. JSON-RPC over stdioをHTTPにブリッジ                  │   │
│  │  5. 状態はRedis/Cloud Storageに永続化                    │   │
│  │                                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**実装の鍵:**

```typescript
// Universal Wrapper の核心部分
type McpCatalogEntry = {
  id: string;
  command: string;
  args: string[];
  envVarKeys: string[];
  stateRequirement: "STATELESS" | "STATEFUL_OAUTH" | "STATEFUL_FILE";
};

// 新規MCPサーバー追加 = カタログにエントリ追加のみ
// コード変更不要
```

**メリット:**
- 新規MCPサーバー追加がDB操作のみで完了
- 統一されたラッパーでメンテナンスコスト低
- スケーラビリティ高

**デメリット:**
- ステートフルMCPサーバーへの対応が限定的
- Cloud Run起動時間のオーバーヘッド

### 2.3 案C: Per-User Container方式

**概要**: ユーザーごとに専用コンテナを起動し、状態を永続化

```
┌─────────────────────────────────────────────────────────────────┐
│                   Cloud Run Jobs / GKE                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ User A Container                                         │   │
│  │   ├── mcp-github (OAuth token: user A)                   │   │
│  │   ├── mcp-slack (OAuth token: user A)                    │   │
│  │   └── Persistent Volume (状態保存)                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ User B Container                                         │   │
│  │   ├── mcp-notion (OAuth token: user B)                   │   │
│  │   └── Persistent Volume (状態保存)                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**メリット:**
- 完全なステートフル対応
- ユーザー間の完全分離

**デメリット:**
- コスト高（ユーザー数 × コンテナコスト）
- スケール管理が複雑
- 起動時間のレイテンシ

### 4.2 Phase 1: Universal MCP Wrapper

**目標**: ステートレスなstdio MCPサーバーを汎用的にホスト

**実装内容:**

```typescript
// 新規ファイル: apps/mcp-wrapper/src/index.ts

type WrapperConfig = {
  mcpServerId: string;
  command: string;
  args: string[];
  env: Record<string, string>;
};

// 1つのCloud Runインスタンスで任意のstdio MCPサーバーを起動
const wrapStdioMcp = async (config: WrapperConfig) => {
  const process = spawn(config.command, config.args, {
    env: { ...process.env, ...config.env },
  });

  // stdio <-> HTTP ブリッジ
  return createHttpBridge(process.stdin, process.stdout);
};
```

**期間**: 2-3週間

### 4.3 Phase 2: Local Agent

**目標**: ステートフルMCPサーバー対応

**実装内容:**

```
tumiki-local-agent/
├── src/
│   ├── agent.ts           # メインエージェント
│   ├── tunnel.ts          # クラウドへのWebSocket接続

```

**接続フロー:**

```
1. ユーザーがLocal Agentをインストール
2. Local AgentがTumiki Cloudに接続 (WebSocket)
3. 認証情報をローカルに保存
4. ステートフルMCPリクエストはLocal Agent経由で処理
```

**期間**: 3-4週間

### 4.4 Phase 3: MCP Catalog + 自動デプロイ

**目標**: 数百〜数千のMCPサーバーを自動対応

**実装内容:**

1. **MCP Catalog DB**
   - npmjs.com / GitHub からMCPサーバーをクロール
   - 自動でカタログエントリ生成

2. **自動デプロイパイプライン**
   - カタログエントリからCloud Run設定を自動生成
   - Terraformでインフラ自動プロビジョニング

**期間**: 4-6週間

## 9. 発展案: 統合アーキテクチャ

### 9.1 案B-1: 個別Cloud Runインスタンス（現状拡張）

**現状の仕組みを維持しつつ、自動化を強化**

```
┌─────────────────────────────────────────────────────────────────────────┐

│  ┌───────────────────────────────────────────────────────────────────┐ │

│  │   headerEnvMappings: [{ header: "X-DeepL-API-Key",               │ │
│  │                         env: "DEEPL_API_KEY" }],                  │ │
│  │   cloudRunUrl: "https://deepl-mcp-xxx.run.app" }                 │ │
│  └───────────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────┬────────────────────────────────┘
                                         │
         ┌───────────────────────────────┼───────────────────────────────┐
         ▼                               ▼                               ▼
┌─────────────────┐          ┌─────────────────┐          ┌─────────────────┐
│ Cloud Run:      │          │ Cloud Run:      │          │ Cloud Run:      │
```

**改善点:**
- カタログDBでサービス一覧を管理

- tumiki-proxyからカタログ参照してルーティング

**コスト:** MCPサーバー数 × Cloud Run最小コスト

## 10. 実装ロードマップ（改訂版）

### Phase 1: カタログDB + tumiki-proxy統合（2週間）

```
目標: 既存のtumiki-mcp-cloudrunをtumiki本体と連携

1. McpCatalogEntry モデルをPrismaに追加
2. tumiki-mcp-cloudrunの設定をDBに同期
3. tumiki-proxyからCloud Run URLへルーティング
4. Manager UIでカタログ表示

成果: 15個のMCPサーバーがtumiki経由で利用可能
```

### Phase 2: Universal Wrapper（3週間）

```
目標: 軽量MCPサーバーを単一インスタンスで処理

1. Universal Wrapper実装（Go or Node.js）
2. プロセスプール管理
3. カタログDBからの動的設定読み込み
4. 軽量MCPサーバーを移行

成果: Cloud Runコスト削減 + 新規MCP追加の簡略化
```

### Phase 3: 自動カタログ収集（2週間）

```
目標: npm/GitHubからMCPサーバーを自動収集

1. npmjs.com APIでMCPサーバーパッケージ検索
2. package.jsonからメタデータ抽出
3. 自動動作確認（sandbox実行）
4. カタログDBへ自動登録

成果: 数百のMCPサーバーを自動対応
```

### Phase 4: Local Agent（オプション、4週間）

```
目標: ステートフルMCPサーバー対応

1. Local Agentアプリケーション開発
2. tumiki-proxyとのWebSocket接続
3. OAuth認証フローの永続化
4. ファイルシステム系MCP対応

成果: 全種類のMCPサーバーに対応
```
